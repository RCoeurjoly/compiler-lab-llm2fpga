# ADR: Validate the RC vertical slice with a deterministic prompt fixture

- Status: accepted
- Date: 2026-07-16

## Context

The frozen V=6, context-8 representative core has a finite token-level input
domain, so exhaustive comparison against frozen PT2E W8A8 can establish its
bounded compute-interface behavior. That test deliberately starts after token
IDs exist and before a human-facing token is reconstructed. It cannot by itself
show that host tokenization, SV invocation, frozen-image selection, and output
detokenization form one working transaction.

Conversely, one successful prompt demonstration cannot replace exhaustive
token-level evidence: it exercises one path through the finite RC domain.

## Decision

Every canonical RC route must additionally pass at least one deterministic,
end-to-end prompt fixture:

```text
fixed prompt -> tiny V=6 host tokenizer -> eight token IDs
             -> testable SV + frozen model image
             -> six raw output codes + token ID
             -> tiny V=6 host detokenizer -> generated token
```

The fixture is durable and reproducible through the repository/Nix workflow.
It records the tokenizer and detokenizer table/version hashes, prompt bytes,
eight input IDs, frozen PT2E/image/SV/tool hashes, the six raw codes, selected
token ID, generated token, declared and observed latency, and pass/fail result.
The expected numerical output is still generated by frozen PT2E W8A8; the host
does not become an independent numerical oracle.

This is a vertical-slice integration gate in addition to—not instead of—the
exhaustive `6^8` token-level RC observable-equivalence gate. Its immediate
demonstration claim is deliberately bounded to one correct generated token at
reproducible latency for the fixed RC.

## Sequencing

DDR3 is not a prerequisite for the first RC vertical slice. First lower the
complete frozen V=6 RC to testable SV, pass its simulation gates, and test the
same complete RC on the FPGA using its baseline frozen-memory implementation.
Only after that board checkpoint may memory externalization become the next
semantic transformation.

The baseline board checkpoint requires a reproducible XC7K480T bitstream and
must capture the vertical-slice fixture's six raw output codes and token ID
from the actual board. The host compares those values against frozen PT2E for
the fixture and records the observed latency. A successful synthesis,
configuration, or `done` signal alone is not a board result. The record
includes bitstream, place-and-route, constraint, host-program, fixture, and
tool provenance sufficient to repeat the transaction.

The baseline V=6 FPGA candidate contains all RC compute and its frozen model
storage as real synthesizable logic or memories. It may not black-box weights,
operators, or substitute a dummy memory service. Memory-blackbox and
operator-blackbox variants remain allowed only as explicitly labelled
diagnostics; they do not satisfy the complete RC-on-FPGA checkpoint.

At that later stage, the image-backed SV fixture must consume the exact packed
byte image and address map intended for the DDR3 driver, produced by one
versioned packer. It may not use a simulator-only convenient representation.
The external-memory route then repeats its image-backed equivalence gate before
the real DDR3 service is introduced on board.

## Consequences

- The tiny V=6 tokenizer is a real, versioned host component rather than an
  untracked testbench convenience.
- A canonical RC route requires both exhaustive compute-interface evidence and
  at least one host-to-SV-to-host transaction.
- Passing the prompt fixture does not establish behavior for arbitrary text,
  full TinyStories vocabulary, multi-token generation, or DDR3 hardware.
- DDR3 work is deliberately deferred until the complete baseline V=6 RC has
  been tested on FPGA; this avoids mixing lowering/compute risk with
  external-memory risk.
- A baseline FPGA claim requires observed logits/token ID from a reproducible
  XC7K480T bitstream, not merely an implementation artifact or completion pin.
- The complete baseline RC claim excludes black-boxed model compute, weights,
  and dummy memory services. Those are diagnostics only.
- The same fixture may then be rerun with the exact packed-image SV service and
  with the real DDR3 service as distinct external-memory checkpoints.
